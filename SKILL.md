# Knowledge Base Manager

## 知识库位置

知识库根目录即当前 workspace 根目录。
路径因设备而异，始终以 workspace 根目录为基准，使用相对路径操作文件。

## 跨平台运行

当用户讨论平台依赖、跨平台迁移、Cursor/CodeBuddy/Codex/Claude Code 适配、普通聊天模型降级运行等问题时，读取根目录 `PLATFORMS.md`。

`PLATFORMS.md` 只负责平台适配和降级策略，不替代本文件中的工作流定义。

## 首次交互

**首次交互协议定义在 `SOUL.md` 的"首次交互协议"章节，必须严格执行。**

简要流程：
1. 实际读取 `SOUL.md` / `_schema.md` / `SKILL.md` / `topics/` 目录
2. 逐项输出自检报告（已读文件、权限边界、当前主题数量列表）
3. 宣告就绪，等待用户指令

**不允许跳过或用记忆替代实际读取。**

## 核心工作流

### 工作流 A：查询知识库

用户询问知识库内容时：

1. 根据关键词搜索 `topics/` 下的文件
2. 读取匹配的 `_overview.md` 和笔记文件
3. 综合多来源信息回答用户问题
4. 标注信息来自哪些来源笔记

### 工作流 B：从截图/文字创建笔记

用户发送截图或粘贴文字时：

1. **识别内容**
   - 截图：识别图中文字、UI 特征（书页排版、知乎界面、微信截图、B站评论等）
   - 文字：分析文本内容和格式

2. **书籍截图增强**（当识别为书页/读书笔记截图时）
   - 从截图中提取书名、作者等信息
   - 在 `_vault/_index.md` 中查找该书是否有电子版（本地路径字段非空）
   - 如果有电子版：读取原文，理解截图笔记对应的上下文，结合原文对用户的笔记做总结和补充
   - 如果没有电子版：仅基于截图内容做总结，并建议用户是否需要通过 `tools/book-finder.md` 获取电子版
   - 最终生成的笔记应包含：用户原始笔记内容 + 基于原文的上下文补充和总结

3. **推断元数据**（向用户确认）
   ```
   来源类型：book / podcast / web / video / article / other
   来源标题：{推断的标题}
   作者：{推断的作者，如可识别}
   标签：{建议的标签}
   ```

4. **匹配主题**
   - 列出 `topics/` 下已有主题
   - 建议归入最匹配的已有主题，或建议新建主题
   - 等待用户确认

5. **生成笔记预览**
   - 根据 `_schema.md` 和对应模板（`_templates/note-{type}.md`）生成完整笔记
   - 填充 frontmatter 所有字段
   - 正文包含提取的核心内容
   - 展示完整预览给用户

6. **写入**（用户确认后）
   - 按命名规范创建笔记文件到对应主题目录
   - 如果是新主题，先创建主题目录和 `_overview.md`、`_discussions.md`
   - 如果是截图，询问是否存入 `_vault/images/` 并更新 `_vault/_index.md`

### 工作流 C：讨论后同步

与用户就某主题讨论完毕后：

1. **主动询问**："要把这次讨论的要点同步到知识库吗？"

2. **整理讨论要点**（如用户同意）
   ```markdown
   ## YYYY-MM-DD 讨论标题

   **触发话题**：...

   **讨论要点**：
   - ...

   **新理解**：...

   **待深入**：
   - ...

   **学习素材**：

   ### 素材标题

   （讨论中 AI 应用户要求提供的结构化参考资料）

   ---
   ```

   **学习素材段落规则**：
   - 回顾对话，收集 AI 应用户要求提供的结构化内容（时间线、对比表格、知识梳理、概念框架、推荐阅读/视听列表等）
   - 保留原始格式（表格、列表等），直接放入
   - 如本次讨论无此类内容，省略该段落

3. **展示预览**，等待用户确认

4. **写入**
   - 追加到对应主题的 `_discussions.md`（新记录插在最上方，`---` 分隔线之前）
   - 如有重大认知更新，建议更新 `_overview.md` 的"个人理解"部分

### 工作流 G：多 Agent 讨论与沉淀

> 与工作流 C 并行存在。C 负责普通聊天后的轻量同步；G 负责正式开启一场多视角深入讨论，最后沉淀为结构化笔记。本工作流使用根目录 `AGENTS.md` 中定义的角色池和发言格式。

**适用场景**：
- 用户明确说"进入讨论"、"多 Agent 讨论"、"多视角分析"、"拉几个角色聊聊"等；
- 用户希望对某个话题深入分析，而非普通问答；
- 话题涉及文学、社会学、心理学、哲学、历史、文化、传播等多个解释框架；
- 用户希望最后将讨论沉淀到知识库。

#### G0：进入讨论

确认用户希望进入多 Agent 讨论模式。用户已明确要求则直接进入；表达含糊时先询问是否启用本工作流。

#### G1：明确讨论话题

识别并确认：
- 主题句（想讨论什么）；
- 核心问题（这次真正要追问的问题）；
- 预期方向（文本细读 / 社会分析 / 认知机制 / 哲学辨析 / 历史背景 / 知识库沉淀）。

#### G2：读取知识库上下文

优先检索后形成简短"讨论上下文包"：
- 相关 topic 的 `_overview.md`；
- 相关正式笔记；
- 相关 `_discussions.md`；
- 必要时检索 `_vault/_index.md`。

全局上下文包应包括：
- 本轮话题可能涉及的主 topic；
- 与话题直接相关的旧讨论或正式笔记；
- 可供不同 Agent 使用的关联笔记清单；
- 未找到直接关联内容时，也要说明检索范围。

> 多 Agent 讨论不凭空开始，须先连接已有知识库内容，减少模型自由发挥。

#### G3：选择 Agent 阵容

读取 `AGENTS.md`，根据话题类型（见 AGENTS.md 第 7 节）选择 3-5 个 Agent，并根据 `AGENTS.md` 第 6 节「Agent 与知识库主题映射」确定每个解释型 Agent 应优先读取的主题。

每次讨论必须满足：
- 至少 2 个解释型 Agent；
- 1 个方法论审稿人；
- 1 个主持人；
- 如需沉淀，讨论结束时追加知识库管理员。

输出阵容时应同时列出角色与主题读取目标，例如：
```markdown
本轮 Agent 阵容：
- 传播学家：优先读取 `communication`
- 社会学家：优先读取 `sociology`
- 认知心理学家：优先读取 `cognitive-psychology`
- 方法论审稿人：检查依据和边界
- 主持人：综合收束
```

#### G4：Agent 首轮发言

解释型 Agent 发言前，必须根据 `AGENTS.md` 第 6 节的主题映射，读取/参考对应 topic 的 `_overview.md`、相关笔记和 `_discussions.md`，寻找对本轮讨论有启发或关联的内容。

每个 Agent 严格按 `AGENTS.md` 第 4 节的 7 段式发言格式输出（其中"我依赖的依据""我解释不了的部分""我建议保守表述为"为硬约束，不得省略）。

在「我依赖的依据」中必须单独列出「知识库关联」：
- 找到关联时，写出具体文件路径，并说明关联点；
- 未找到直接关联时，明确说明「未找到直接关联笔记」；
- 不得引用未实际读取或未在上下文中出现的笔记。

#### G5：方法论审查与主持人综合

- **方法论审稿人**检查：证据不足、概念混用、过度解释、类比跳跃、需回到原文/材料核实处；至少指出一个风险或边界。
- **主持人**整理：共同点、主要分歧、最有价值的解释路径、最不稳的判断、下一轮追问方向。主持人不新增观点。

#### G6：单次总结或多轮延展

询问用户选择：
- 单次总结（话题较小、已有明确结论）；
- 多轮延展（话题复杂、分歧明显、出现新问题）；
- 回到原文、材料或旧笔记继续检索。

多轮延展时，每轮只保留 2-3 个最相关 Agent，避免讨论膨胀。

#### G7：知识库沉淀

如用户希望记录，启用知识库管理员，判断：
- 推荐归入哪个 topic；
- 写入 `_discussions.md` 还是发展为正式笔记；
- 哪些内容应标记"待验证"；
- 哪些内容不建议沉淀。

沉淀模板（在普通讨论模板基础上保留"不同视角的张力"）：

```markdown
## YYYY-MM-DD 多 Agent 讨论：标题

**触发话题**：...
**核心问题**：...
**启用 Agent**：文本细读者 / 哲学概念分析者 / 方法论审稿人 / ...

**相关知识库上下文**：
- `topics/.../_overview.md`：...
- `topics/.../_discussions.md`：...
- `topics/.../note-*.md`：...

**本轮新增连接**：
- 这次讨论把 A 主题和 B 主题连接起来：...

### 各视角要点
#### {角色名}
- ...

### 主持人综合
**较稳妥的判断**：...
**有启发但需保留的判断**：...
**需要回到原文或材料核实的地方**：...

### 新理解
- ...

### 待深入
- ...

### 可发展为正式笔记的方向
- ...

---
```

**写入前必须展示预览并等待用户确认**（遵循下方权限规则）。

### 工作流 F：每日知识库讨论（自动化触发）

> **触发方式**：由 WorkBuddy 自动化任务每日早上定时执行，也可手动触发。

1. **读取知识库结构**
   - 读取 `SOUL.md` 和 `SKILL.md`，了解知识库结构和行为规范
   - 扫描 `topics/` 下所有主题目录

2. **读取主题内容**
   - 读取各主题的 `_overview.md` 和笔记文件

3. **读取讨论历史**
   - 读取各主题的 `_discussions.md`
   - 记录"待深入"列表和近期已讨论的主题
   - **原则：每次选题与上一次不同，维持多样性**

4. **搜索外部热点**（通过以下 Skill/工具）
   - TrendingNow、腾讯新闻、每日资讯、Tavily Search、微信公众号搜索
   - 目的：找到热点与笔记内容的交叉或对立点

5. **选择切入点**，优先顺序：
   - 不同主题 / 不同来源笔记之间能产生有趣对话的交叉点
   - 某条笔记中的观点可被质疑、延伸或与其他学科关联的地方
   - 热点内容与笔记的关联点（质疑或延展）
   - `_discussions.md` 中"待深入"列出但还没讨论过的问题

6. **分层提出切入点**
   - **第一层（必须）**：用 1-2 句日常语言说出这个切入点，口吻像朋友聊天，不要上学术框架，不要铺垫背景，直接抛有趣的点
   - **第二层（备用）**：在第一层后加一行小字：「如果你想深入聊，可以问我 XXX」——给钩子，不强迫

7. **由用户决定深度**
   - 发出后等待回应
   - 用户随口回应 → 轻松聊
   - 用户追问或说"展开" → 才进入深度分析
   - **不主动堆砌理论框架**

8. **讨论结束后**
   - 按照工作流 C 的规范，询问用户是否要同步讨论要点到知识库

### 工作流 D：管理原始材料

本工作流只处理 `_vault/` 原始材料层，不自动写入 `topics/` 正式笔记。

#### D1：存入原始文件

用户需要存入原始文件时：

1. 确认文件类型，决定存入 `_vault/` 的哪个子目录
2. 分配新的 ID（读取 `_vault/_index.md` 获取当前最大 ID，递增）
3. 生成映射表新行预览
4. 用户确认后追加到 `_vault/_index.md`

#### D2：加工 vault 原始材料

用户需要让 LLM 更方便读取 `_vault/books/` 中的 PDF / EPUB / MOBI，或要求“给 PDF 做体检”“转 Markdown”“生成 LLM 可读副本”“生成质量报告”时：

1. 读取 `tools/vault-ingest.md`，遵守其中的边界和命令说明
2. 原始文件保持在 `_vault/books/`，不得覆盖、移动或删除
3. 派生文件统一写入：
   ```text
   _vault/extracted/books/
   ```
4. 每个成功提取的材料应生成：
   ```text
   {book-name}.md
   {book-name}.meta.json
   {book-name}.report.md
   ```
5. PDF / EPUB 转 Markdown、质量报告、OCR 判断都属于材料层处理，不得自动修改：
   ```text
   topics/
   topics/*/_overview.md
   topics/*/_discussions.md
   topics/*/note-*.md
   ```
6. 扫描版 PDF 或坏文字层 PDF 默认只标记 `needs_ocr`，不得自动 OCR；OCR 需用户明确确认后再执行
7. Markdown 是 LLM 友好阅读副本，不是权威原文；精确引用、页码、术语判断必须回看 `_vault/books/` 中的原始材料
8. 如果用户后续希望将阅读理解沉淀为正式笔记，必须另走工作流 B 或工作流 C，先展示预览，等待用户确认

当前脚本入口：

```bash
python tools/scripts/vault-ingest/vault_ingest.py inventory
python tools/scripts/vault-ingest/vault_ingest.py inspect "_vault/books/xxx.pdf"
python tools/scripts/vault-ingest/vault_ingest.py inspect-all
python tools/scripts/vault-ingest/vault_ingest.py extract "_vault/books/xxx.pdf"
python tools/scripts/vault-ingest/vault_ingest.py extract-all --dry-run
python tools/scripts/vault-ingest/vault_ingest.py extract-all
```

## 权限提醒

严格遵循 `SOUL.md` 中的权限规则：

- **读取**：自由执行
- **写入/新增**：必须先预览，用户确认后才执行
- **删改已有内容**：禁止（即使用户口头说"随便改"）
- **删除文件**：禁止

如用户要求删除或修改已有内容，提醒用户手动操作。此规则无例外。

## 文件命名速查

| 类型 | 命名格式 |
|------|---------|
| 主题目录 | `topics/{english-slug}/` |
| 主题总览 | `_overview.md` |
| 讨论记录 | `_discussions.md` |
| 书籍笔记 | `note-book-{slug}.md` |
| 播客笔记 | `note-podcast-{slug}.md` |
| 网络笔记 | `note-web-{slug}.md` |
| 视频笔记 | `note-video-{slug}.md` |
| 文章笔记 | `note-article-{slug}.md` |
| 散记/杂记 | `note-misc-{slug}.md`（source_type: other，多来源碎片合集） |
| 通用笔记 | `note-{slug}.md` |
